Three concrete steps

  • Define a fixed vocabulary of discrete symbols.
  • Split input text into pieces from that vocabulary.
  • Map each piece to an integer ID.

Tokenization
Word-Level

Approach 1: word-level tokenization — simple, but brittle

Running sentence under word-level tokenization

If the vocabulary contains every word:
[researchers] [fine-tune] [large-scale] [models]
But compounds like fine-tune and large-scale are often missing. They collapse to unknown tokens:
[researchers] [UNK] [UNK] [models]

What works

  • Short sequences for familiar text.
  • Common words stay intact as single units.

Why it breaks

  • Language is open-vocabulary. English Wikipedia alone has ~1.5M unique word forms.
  • New terms like "GPT-4", "LoRA", and "RLHF" create unknowns.
! Word-level tokenization is efficient, but brittle. Anything outside the fixed vocabulary becomes unknown.
Tokenization
Character-Level

Approach 2: character-level tokenization — flexible, but expensive

Running sentence under character-level tokenization

[r][e][s][e][a][r][c][h][e][r][s] [ ] [f][i][n][e][-][t][u][n][e] [ ] [l][a][r][g][e][-][s][c][a][l][e] [ ] [m][o][d][e][l][s]
Word-level (best case)
4 tokens
Character-level (same sentence)
~40 tokens
Scaling: 500-token passage
~2,500 character tokens

What works

  • No unknown tokens — ever.
  • Handles new words, typos, code, any language.

Why it gets expensive

  • Sequences become \(5\text{–}10\times\) longer.
  • Self-attention is \(O(n^2)\), so \(5\times\) longer means roughly \(25\times\) more compute.
  • The model spends capacity reassembling words from letters on every forward pass.
! Character-level coverage is perfect, but too expensive. Quadratic attention scaling makes it impractical at real sequence lengths.

Subword Strategy

Approach 3: subword tokenization — the practical default

Running sentence under subword tokenization

[research] [ers] [fine] [-] [tune] [large] [-] [scale] [model] [s]

Frequent stems become single tokens. Hyphens, prefixes, and suffixes are their own reusable pieces — no word is ever fully unknown.

Word-level

4 tokens

Short, but brittle. Breaks on hyphens, new jargon, and multilingual text.

Character-level

~40 tokens

Full coverage, but very long. Quadratic attention cost makes it impractical.

Subword

~10 tokens

Near-word efficiency with no unknown tokens. English prose averages about 0.75 words per token.

+ Subword gives near-word efficiency with broad coverage. That is why it became the default in modern LLMs.
i One tradeoff remains: characters inside the same token chunk are no longer individually visible to the model. We will return to that at the end of this section.
Tokenization
BPE

How does the tokenizer learn these pieces?

Subword pieces are not hand-written rules. They are learned from data using Byte Pair Encoding (BPE).

Start: characters
Count adjacent pairs
Merge most frequent
Repeat N times

The core rule

Start from characters. Repeatedly merge the most frequent adjacent pair in the corpus. After many merges, those reusable pairs become the vocabulary.

Why it works

BPE is driven by frequency, not grammar. It finds stems, prefixes, and suffixes because those strings appear together often in real text.

i BPE is a frequency-based compression algorithm. It builds the vocabulary the data demands, not the one a linguist would design.
? Now let's run BPE on a tiny corpus and watch the merges happen.

Tokenization
Why It Matters

Why tokenization constrains reasoning

The mechanism

The model computes on token vectors, not character vectors. If one token covers many letters, character-level reasoning has to be reconstructed from subword pieces.

"strawberry" → [straw] [berry] 2 token vectors. No separate letter vectors.

Cosma et al., "The Strawberry Problem: Emergence of Character-level Understanding in Tokenized Language Models", EMNLP 2025.

Where this gets hard

  • Counting specific letters in a word.
  • Extracting the nth character from a string.
  • Detecting repeated characters.
  • Reversing a string exactly.

Important nuance

Tokenization is a structural disadvantage, not a hard ceiling. Character-level behavior can emerge late in training, and small architectural changes can improve it.

+ Tokenization defines the vocabulary, shapes sequence length, and sets the granularity of every computation that follows.

Embeddings
Key Properties

What embeddings give you, and what they still lack

What embeddings already give you

  • Learned table \(E \in \mathbb{R}^{|V| \times d_{\text{model}}}\): one trainable row per token type.
  • Often one of the largest parameter blocks, scaling with \(|V| \times d_{\text{model}}\).
  • Outputs \(X^{(0)}\) already match model width — later blocks consume them directly.
  • Useful structure appears as directions, neighborhoods, and relative offsets.

What later layers still have to add

  • Sense is unresolved: one row can support multiple intended meanings.
  • No token-token interaction yet — rows cannot use neighboring words.
  • Row identity alone does not specify sentence role or dependency structure.
  • Later layers must refine these vectors into context-dependent states.
→ Next: look at the same vectors through different semantic projections, then show why context is needed to pick the intended meaning.

What depth means
Each layer runs the same transformer block again. The sequence shape stays fixed; the representation gets more refined.
One block = one refinement round — attention and FFN read the current stream and write back an update.
The next block starts from the improved stream — later layers continue from what earlier layers already built.
Depth buys repeated computation — the model gets multiple chances to combine and refine information.
