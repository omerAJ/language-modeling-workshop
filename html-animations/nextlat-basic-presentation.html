Latent Execution Objective for Code LLM Training

Brief: add a world-model objective to improve execution reasoning while preserving normal code generation behavior.

Project objective

Increase real code correctness (not just syntax fluency) by teaching the model what each line does to program state.

Current scope

Start with a focused 7B LoRA pilot, then expand only if results show clear value.

Pilot-first approach: validate signal quickly, then consider full fine-tuning as a follow-up phase.

Why This Project

Standard autoregressive (AR) training is strong at token prediction, but it does not explicitly train state transition understanding.

Current pain

  • Model often writes valid-looking code that fails during execution.
  • Logic errors persist even when syntax is correct.

Hypothesis

  • If we train line-to-state transitions, logic consistency should improve.
  • We can do this without changing inference behavior.

Motivation from Recent Work

This approach combines ideas from LLM-JEPA, NextLat, and CWM-style code world modelling.

LLM-JEPA + NextLat intuition

  • LLM-JEPA: predict one view of data from another view in latent space.
  • NextLat: treat token sequence states as evolving belief states where the final state predicts what comes next.

From CWM to this method

  • CWM frames code world modelling as predicting the local variable dict and how each line changes it, typically autoregressively token-by-token.
  • Here we avoid token-level state generation and predict the state transition directly in latent space.
  • Inspired by JEPA, token-level reconstruction can include redundant details not needed for transition learning.

Target objective: learn delta_state_latent for each action line, not autoregressive state-dict text prediction.

Core Idea in One Slide

Keep normal AR code training, but replace token-by-token state dict prediction with latent delta prediction for state changes.

Input trace step context_t, action_t, state_before_t, state_after_t
Read latents gather CTX_SUM, ACT_SUM, and current STATE_SUM from the same transformer
Predict delta transition head predicts latent state change caused by the action line
Train objective compare predicted next-state latent vs detached target next-state latent
Inference unchanged No token-level state generation Locals-only in v1

Concrete Data Example (Trace + Tokens)

Function call and traced locals

def add_if_positive(x, y):
    total = x + y
    if total > 0:
        out = total * 2
    else:
        out = 0
    return out

if __name__ == "__main__":
    result = add_if_positive(2, 5)

# Trace inside add_if_positive frame
step 0 (function entry from main call):
state_before_t0 = {}
state_after_t0  = {"x": 2, "y": 5}

step 1:
state_before_t1 = {"x": 2, "y": 5}
action_line_t1  = total = x + y
state_after_t1  = {"x": 2, "y": 5, "total": 7}

Model inputs with summary tokens

# Step 1 training sample
<CODE_CTX>... result = add_if_positive(2,5) ... total = x + y ...</CODE_CTX> <CTX_SUM>
<ACTION>total = x + y</ACTION> <ACT_SUM>

<STATE>{"x":2,"y":5}</STATE> <STATE_SUM>
<STATE>{"x":2,"y":5,"total":7}</STATE> <STATE_SUM>

Pass A reads CTX_SUM/ACT_SUM; Pass B/C read STATE_SUM. We train against latent transition targets, not token-by-token state dict output.

Simple Architecture Diagram (Latent Transition)

Concatenate CTX_SUM, ACT_SUM, and STATE_SUM_t, predict a latent delta, then apply a residual update for predicted STATE_SUM_t+1 (instead of autoregressive state text decoding).

pred_state_t+1 = h_state_t + f([h_state_t, h_ctx, h_act])

Why This Is Low-Risk to Existing Capability

Preserve baseline behavior

  • AR loss remains active on code tokens.
  • Summary tokens are excluded from AR loss.
  • No AR loss on state-only passes.

Controlled rollout knobs

  • lambda controls WM strength.
  • Optional delayed WM start (AR warmup first).
  • No-op/state-change ratio is tunable.

Fallback path is simple: reduce WM weight or disable WM path and keep AR training pipeline intact.

Pilot Design (Built for Causal Attribution)

All variants train/evaluate on the same dataset and protocol.

  1. Base 7B model (no new tuning)
  2. 7B LoRA AR-only (control for dataset/training exposure)
  3. 7B LoRA AR + WM (proposed method)

What this isolates

Improvement from extra data vs improvement from the new latent objective.

v1 training mode

Teacher-forced WM path using ground-truth executed action lines from traces.

Data and System Work Needed

Data pipeline

  • Extract step tuples from traced execution.
  • Deterministic locals serializer (types, order, truncation).
  • Validation checks and no-leak train/val/test splits.

Training stack changes

  • Add summary tokens: <CTX_SUM>, <ACT_SUM>, <STATE_SUM>.
  • Implement 3-pass latent readout + transition head.
  • Combine losses: L_total = L_AR + lambda * L_WM.

Success Metrics and Evaluation Criteria

Must hold

  • Coding quality does not regress vs AR-only control.
  • Pass@1 and execution success stay stable or improve.

Must improve

  • Latent transition error improves on validation traces.
  • Logic-error rate trends better than syntax-only gains.

Positive signal: AR+WM improves execution-quality indicators with stable coding quality. Weak signal: regression or no measurable value vs AR-only.

Main Risks and Mitigations

Risk: noisy traces hurt training

Mitigation: strict dataset validation, serialization checks, malformed-sample filtering.

Risk: WM objective interferes with AR

Mitigation: delayed WM start, conservative lambda, and AR-only control monitoring.

Risk: LoRA underestimates full potential

Mitigation: treat LoRA result as signal test; escalate to full FT only if pilot is positive.

Risk: unclear attribution of gains

Mitigation: same dataset + same eval protocol across base, AR-only, and AR+WM.
