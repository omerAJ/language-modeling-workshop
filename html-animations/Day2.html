Tokenization

What tokenization actually does

Text is messy and open-ended; the model needs a fixed interface.

Running sentence (we will reuse this across tokenization types)
"Our startup launched slowerify today."
Raw text
Tokenizer
Token IDs
Embeddings

Three concrete steps

  • Define a vocabulary of discrete symbols.
  • Split text into token pieces from that vocabulary.
  • Map each token piece to an integer ID.

Same sentence, now as model input

[Our] [startup] [launched] [slower] [ify] [today] [.]
[412, 9621, 7710, 28904, 771, 1908, 13]

IDs are illustrative; exact values depend on the tokenizer vocabulary.

i Tokenization chooses the units of computation. Next we compare word-level, character-level, and subword-level on this exact sentence.
Tokenization
Word-Level Baseline

Word-level on the same sentence: simple, but brittle

Running sentence under word-level tokenization

[Our] [startup] [launched] [slowerify] [today] [.]
If slowerify is unseen in the fixed word vocabulary, it may collapse to an unknown token:
[Our] [startup] [launched] [UNK] [today] [.]

What is attractive

  • Easy mental model: one word = one token.
  • Short sequence for this sentence (~6 tokens).
  • Feels natural to human readers.

Why it breaks

  • Language is open-vocabulary: new names, typos, and jargon keep appearing.
  • A fixed vocabulary becomes huge, or drops rare words as unknowns.
  • The model cannot reuse sub-parts like stems or suffixes for unseen words.
! Word-level tokenization is efficient for familiar words, but brittle whenever language invents something new.
Tokenization
Character-Level Baseline

Character-level on the same sentence: flexible, but long

Running sentence under character-level tokenization

[O] [u] [r] [ ] [s] [t] [a] ...
Word-level (same sentence)
~6 tokens
Character-level (same sentence)
~37 tokens (includes spaces/punctuation)

Exact counts vary by tokenizer conventions, but the sequence-length gap is the key point.

What is attractive

  • No unknown words at all.
  • Handles arbitrary spelling and novel strings.
  • Natural for character-sensitive tasks.

Why it is not default

  • Sequences become much longer.
  • Transformer cost grows with sequence length.
  • The model spends compute rebuilding words from tiny pieces.
! Character-level is flexible, but it makes the model do too much work on very small pieces.

Subword Strategy

Subword on the same sentence: practical middle ground

Running sentence under subword tokenization

[Our] [startup] [launched] [slower] [ify] [today] [.]

Frequent chunks become single tokens, and rare words can still be composed from smaller reusable pieces.

Word-level

Short sequence, but brittle on unseen words.

Character-level

Always flexible, but very long sequence.

Subword-level

Near-word efficiency with far better flexibility.

+ Subword tokenization memorizes common chunks and composes uncommon ones.

This is why subword tokenization became the practical default in modern LLMs.

Tokenization
BPE

Byte Pair Encoding (BPE): the canonical subword algorithm

Start from small units
Count adjacent pairs
Merge most frequent pair
Repeat

Core rule

BPE repeatedly merges the most frequent adjacent pair into a new token. Over many merges, common fragments become reusable subword units.

Important framing

BPE is driven by frequency, not meaning. It may discover meaningful-looking pieces, but only because those strings appear often.

i BPE is a frequency-based compression scheme turned into a text interface.

Tokenization
Why It Matters

Why tokenization influences letter-level behavior

Concrete mechanism (not random)

Character-level question: "How many r letters are in slowerify?"
Tokenizer interface first: [slower] [ify] (or another subword split)

Attention/FFN computations run on token vectors. For letter-level tasks, the model often has to reconstruct character structure indirectly from token pieces.

1. Input interface

Prompt arrives as token IDs, not raw character arrays.

2. Core computation

Transformer layers operate on token embeddings and token states.

3. Extra reconstruction

Letter-level answers may require rebuilding characters across token chunks.

Tasks where this intuition helps

  • Counting letters in a word.
  • Reversing a word exactly.
  • Extracting the nth character.

Important nuance

Tokenization is a contributor, not the only cause. Training data, optimization, and decoding strategy also affect these outcomes.

! The model is fluent in token pieces first, and only indirectly aware of letters.

What tokenization gave us

[412, 9621, 7710, 28904, 771, 1908, 13]

These integers are discrete labels or addresses in a vocabulary table.

Why this is not enough

  • ID numbers by themselves do not encode semantic similarity.
  • Arithmetic on raw IDs is meaningless for language computation.
  • The model needs dense vectors it can compare, mix, and transform.
→ Now we have token IDs. The next question is: how does the model turn those discrete IDs into something it can compute with? That is the embedding layer.
